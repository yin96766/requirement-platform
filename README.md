# 需求管理平台 MVP

一个可直接打开运行的前端原型，用于统一收集、梳理、跟踪、评审与落地业务需求、产品需求及项目需求。

## 功能范围

- 需求全生命周期看板：`待评审`、`设计中`、`开发中`、`已阻塞`、`已上线`、`已归档`
- 需求识别转换：将自然语言输入转换为产品和研发可执行的结构化需求说明
- 文档导入识别：支持导入 `txt`、`md`、`csv`、`json`、`docx`、`xlsx`、`xls` 并自动拆分多个需求
- 批量生成需求单：支持从单次识别结果中直接生成多个需求单
- 导入历史记录：自动保存成功导入的文档记录，支持查找、恢复和删除
- 需求看板：按标题、部门、负责人、类型、状态、优先级筛选
- 评审中心：待评审队列、评审结论沉淀
- 落地推进：计划上线时间轴、审计日志
- 状态流转：支持拖拽卡片变更状态并自动记录日志
- 数据持久化：使用浏览器 `localStorage` 保存数据

## 使用方式

1. 直接在浏览器中打开 [index.html](C:\Users\yinch\Documents\New project\index.html)
2. 在“需求识别”中输入自然语言，生成结构化需求说明
3. 或导入 `txt`、`md`、`csv`、`json` 文档，自动识别并拆分多个需求
4. 点击“生成需求单”或“批量生成需求单”创建需求
5. 在“需求看板”中筛选、编辑需求
6. 在“总览看板”中拖动卡片调整需求状态
7. 点击“重置示例数据”恢复初始样例

## 对外访问

如果你想让别人通过网址访问这个平台，当前项目最适合发布到 GitHub Pages，因为它是纯静态前端页面。

1. 新建一个 GitHub 仓库并把当前项目推上去
2. 仓库中已经包含 [.github/workflows/pages.yml](C:\Users\yinch\Documents\New project\.github\workflows\pages.yml)
3. 在 GitHub 仓库设置中打开 `Settings -> Pages`
4. 将 `Source` 设为 `GitHub Actions`
5. 推送代码后，GitHub 会自动发布页面

发布后的默认项目地址通常是：

- `https://<你的 GitHub 用户名>.github.io/<仓库名>/`

注意：

- 这是“共享应用地址”，不是“共享浏览器本地数据”
- 当前需求、导入历史等数据保存在访问者自己浏览器的 `localStorage`
- 别人在他自己的电脑上打开网址时，会看到同一套系统界面，但不会自动看到你电脑里的数据
- 如果你要“多人看到同一份需求数据”，下一步必须接后端数据库和账号体系
